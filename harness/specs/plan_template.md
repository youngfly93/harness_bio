## Spec 质量检查清单

Agent 逐项检查，全部通过才可开始执行：

```
□ 成功标准全部可衡量（非模糊描述，如"做得好"无效）
□ 固定参数全部明确（无"合适的"、"一般的"、"常用的"等模糊词）
□ 每个 Step 都有明确的 输入 / 输出 / QC检查点
□ 比较组定义无歧义（Treatment vs Control 方向明确）
□ 交付清单文件名和格式已确定
□ 物种和注释库已指定
□ 数据来源可访问（GEO编号存在 / 文件路径有效）
```

**检查结果处理**：
- 全部通过 → 进入执行
- 有缺失项 → 列出缺失清单，等待补充
- 参数与领域惯例矛盾 → 标记警告，询问确认

## QC 断言语法

每个 Step 中使用 `- QC: key operator value` 格式定义机器可读的验收检查。
`validate.sh` 会自动解析这些行，从 `numeric_reference.tsv` 查找实际值并校验。

**操作符**：

| 操作符 | 含义 | 示例 |
|--------|------|------|
| `==` | 等于 | `- QC: sample_total == 12` |
| `>` | 大于 | `- QC: gene_count > 10000` |
| `>=` | 大于等于 | `- QC: mapping_rate >= 0.8` |
| `<` | 小于 | `- QC: na_ratio < 0.05` |
| `<=` | 小于等于 | `- QC: duplication_rate <= 0.3` |
| `~` | 范围（含两端） | `- QC: deg_total ~ 50-5000` |
| `exists` | 文件存在 | `- QC: results/deg.csv exists` |

**规则**：
- key 使用 snake_case，需与 `numeric_reference.tsv` 中的 key 一致
- `exists` 操作符的 key 是相对于项目目录的文件路径
- `?` 表示待填写的占位符，scaffold 生成后必须替换为具体值
- 旧格式 `- QC检查点：` 仍可保留为自由文本备注，但不会被自动解析
