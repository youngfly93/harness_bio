# 五维质量审计体系

> 每个项目交付前必须通过全部五个维度。
> 使用 bio-result-auditor sub-agent 执行自动审计。
> 审计记录统一写入 `execution_log.md`。
>
> **输出格式**：审计结果必须输出为 JSON，schema 见 `audit_schema.json`，示例见 `audit_example.json`。
> 结构化输出使 audit→fix→re-audit 循环可机械化解析，无需人工阅读自然语言。

---

## 五个维度（分层审计）

| 维度 | 检查内容 | 执行者 | 判定标准 |
|------|---------|--------|---------|
| **完整性** | 每个 Step 有产出；交付清单文件存在且非空 | `validate.sh`（脚本） | 缺一个文件 = FAIL |
| **数据准确性** | 数值一致；QC 断言通过 | `validate.sh`（脚本） | 一个不一致 = FAIL |
| **方法合理性** | 统计检验、校正、归一化、阈值 | `/codex:rescue`（交叉验证） | 方法不当 = FAIL |
| **图表质量** | 白底、标签、配色、分辨率、legend | `/codex:rescue`（交叉验证） | 任一不达标 = FAIL |
| **交付规范** | 命名、结构、AI痕迹 | `ai_scan.sh`（脚本） | AI 残留 = FAIL |

**为什么分层**：完整性/准确性/交付规范是可机械化检查的 → 用脚本更快更准。
方法合理性/图表质量需要领域判断 → 用第二个 Agent（Codex/GPT-5.4）交叉验证，
避免"自己批改自己作业"的共享盲区。若 Codex 不可用，退化为 WARN。

## 触发规则

### 必须触发

| 时机 | 审计类型 |
|------|---------|
| 每个 Step 完成后 | 该 Step 的局部审计（完整性 + 数据准确性） |
| 全部 Step 完成后 | **五维全面审计** |
| 报告生成后 | 数值交叉校验审计 |
| 打包前 | 交付规范审计 + AI痕迹扫描 |

### 可选触发

| 时机 | 审计类型 |
|------|---------|
| 修复 bug 后 | 回归审计（确认没引入新问题） |
| 方法变更后 | 方法合理性审计 |

## 审计循环协议

```
轮次上限：5 轮
目标：≤ 2 轮全部 PASS（当前平均 3-4 轮）

Round N:
  1. 运行 validate.sh + ai_scan.sh → 完整性/准确性/交付（脚本）
  2. 运行 /codex:rescue → 方法合理性/图表质量（交叉验证）
     └─ Codex 不可用时 → 这两个维度标 WARN，不阻断
  3. 汇总五维结果 → audit_result.json
  4. IF 全部 PASS → 进入交付阶段
  5. IF 有 FAIL:
     a. 按维度列出失败项
     b. Claude 执行修复（Codex 不改文件）
     c. 进入 Round N+1
  6. IF Round > 5 → 停下来，上报人工介入

每轮审计结果记录到 execution_log.md：
| 轮次 | 结果 | 失败维度 | 修复内容 |
```

## 图表审查协议

对每张图表逐项检查，详细规则见 `harness/quality/fig_review.md`：

```
□ 图表类型适合展示的数据（不用饼图展示连续变量）
□ 坐标轴标签完整（含单位）
□ 图例清晰，分组标识明确
□ 配色区分度足够且视觉舒适（非彩虹色、非高饱和纯色）
□ 标题/描述与图表实际内容一致
□ 统计标注正确（*p<0.05, **p<0.01, ***p<0.001）
□ 白底、无网格线
□ 分辨率 ≥ 300 dpi
□ 文字大小在打印后可读
```
